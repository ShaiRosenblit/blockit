# Lens 1 — Repurpose an existing primitive (Opus)

The thesis: Puzzle's win was that line clears — a *reward* in Classic — became the player's *strategic deletion verb*. The same primitive carries two opposed meanings depending on context, and the player must hold both in mind every placement.

The eight candidates below each pick a different existing primitive (a tray piece's footprint, single-vs-combo line clears, the rotation tap, the row/column axis distinction, color, tray refill timing, the placement event itself, the cell-count of consecutive pieces) and force it to do double duty — load-bearing, not decorative. Each was screened against Mirror / Breathe / Pipeline / Scar before being kept.

---

## Candidate L1-1 — Starve: tray pieces double as anti-templates whose footprint must be unfittable

- Lens: 1 (Repurpose an existing primitive)
- Generated by: Claude Opus
- Rule changes:
  - Tray on round-start contains `K` placeable pieces (drag-to-board as usual) and `S` **starve pieces** (visually marked with an X / lock overlay, displayed in a fixed sidebar slot — generator picks `S = 1..2`).
  - Starve pieces cannot be dragged onto the board. They are pure constraints.
  - No tray refill: this is a finite Puzzle-style instance.
  - All other Classic placement / row+column clear rules unchanged.
- Win condition: `placeableTray.length === 0` AND for every starve piece `s` in starveTray: `enumerateLegalPlacements(board, s) === ∅` (tested across all 4 rotations).
- Lose condition: at least one placeable piece in `placeableTray` has zero legal placements on the current board (stuck).
- New question per placement: "After this placement (and any clears it triggers), does any 4-connected empty region of the board still admit a fit for any starve piece's silhouette?" Classic/Puzzle never ask this — they only check fit for pieces you intend to place.
- Dual-purpose mechanic: **piece footprint**. A piece is simultaneously (a) something the player wants to drop into the board to make progress and (b) a forbidden silhouette: the empty space must contain no congruent fit for the starve subset. Same primitive (the cells-array of a `PieceShape`), two opposed senses.
- Antagonist: the starve pieces. They are concrete on-board UI elements (sidebar tiles) whose silhouettes the player must imprison. The "defeat" verb is "deny it any home on the resulting board."
- Negative-space constraint: yes — load-bearing. Empty regions must be sculpted small enough or jagged enough that no rotation of any starve piece fits anywhere.
- Order coupling: line clears change which empty regions exist; a row clear creates a wide horizontal empty band that re-admits starve fits. Placing the same `K` placeables in different orders lands different starve-fit verdicts.

## Candidate L1-2 — Combo Lock: line clear is hardened — single clears delete only player-placed cells, combo (≥2 lines) clears delete pre-fill too

- Lens: 1 (Repurpose an existing primitive)
- Generated by: Claude Opus
- Rule changes:
  - Pre-fill cells are tagged with a sentinel color (existing pattern; reuse `BLOCKER_COLOR`).
  - `applyPlacementAndClear` is replaced: detect lines as today, but if the placement triggers exactly 1 line clear, only cells whose color is *not* the pre-fill sentinel are erased; pre-fill cells in the cleared line **stay** (the row is full, then partially re-emptied to just the pre-fill cells).
  - If the placement triggers ≥2 simultaneous line clears, the standard full clear runs (pre-fill included).
  - Puzzle-style finite tray and target-pattern win condition.
- Win condition: `boardMatchesTarget(board, target)` after final placement.
- Lose condition: tray non-empty AND `hasValidMoves(board, tray) === false`.
- New question per placement: "Is this completion arriving alone (single-clear, pre-fill survives) or coinciding with another row/column also closing this turn (combo, pre-fill dies)?" The player must hold completed lines back, mid-row, until a second axis can close on the same placement.
- Dual-purpose mechanic: **line clear**. Same primitive, two senses: as a single event it is a *soft sweep* that only erases your own fills (useful for resetting tray-noise rows but powerless against pre-fill); as a combo it is a *hard sweep* that also kills pre-fill. The player wants single clears to keep playing space open, but fears single clears because they consume a row that could have been combo-paired against pre-fill.
- Antagonist: pre-fill cells whose only exit is a combo. The generator guarantees ≥1 pre-fill cell that lies on a line that single-clears can complete but only combos can erase.
- Negative-space constraint: yes — rows that intersect pre-fill must be left partially empty until a column on the same placement is also primed; filling them prematurely "burns" the row.
- Order coupling: filling a row R's last gap on move 4 is materially different from filling it on move 5 if move 5 also closes column C. Same set of pieces in different orders ⇒ different combo opportunities ⇒ different surviving pre-fill ⇒ different end-board.

## Candidate L1-3 — Gear: every rotation tap rotates the tray piece AND a fixed 3×3 pivot region of the board

- Lens: 1 (Repurpose an existing primitive)
- Generated by: Claude Opus
- Rule changes:
  - The board has a fixed, marked **pivot region** at rows 2..4 × cols 2..4 (visible via a faint inset border on those 9 cells; no new component).
  - When the player taps a tray piece to rotate it 90° CW, the contents of the pivot region also rotate 90° CW about the pivot's center cell (a deterministic 9-cell permutation; cells inside the pivot region — pre-fill, placed cells, or empty — all participate).
  - Placement, clears, target match all unchanged otherwise.
  - The pivot rotation is purely geometric within those 9 cells: it never propagates outside the 3×3 patch.
  - Puzzle-style finite tray and target.
- Win condition: `boardMatchesTarget(board, target)`.
- Lose condition: tray non-empty AND `hasValidMoves(board, tray) === false` after current rotations.
- New question per placement: "Is the rotation count needed for this piece to *fit* the same as the rotation count needed to put the pivot region's pre-fill where row/column clears can reach it?" If not, which one do I sacrifice?
- Dual-purpose mechanic: **rotation tap**. The tap is simultaneously a piece-orienting verb (existing) and a board-sculpting verb (new). Player wants taps (pivot pre-fill rearrangement) and fears taps (piece misorients away from a needed fit).
- Antagonist: pre-fill seeded into the pivot region in a configuration whose *only clearable orientation* is, e.g., 2 rotations away from its initial orientation. The player must accumulate 2 net rotations across all tap events before that pre-fill can be cleared.
- Negative-space constraint: yes — empty cells inside the pivot region also rotate, so an empty L that the player wanted in the bottom-left of the patch may end up top-right after the tap, which can poison subsequent placements.
- Order coupling: tap-tap-place, tap-place-tap, and place-tap-tap with the same piece set yield different pivot final states; pivot state interacts non-commutatively with the line-clear pipeline.

## Candidate L1-4 — Grain: row clears delete; full columns do not clear and persist as dead weight until a row clear breaks them

- Lens: 1 (Repurpose an existing primitive)
- Generated by: Claude Opus
- Rule changes:
  - `detectCompletedLines` runs as today, but `clearLines` only erases the row entries; column entries are dropped from the cleared set.
  - A column reaching all-8-filled does NOT trigger any clear and the cells stay; the column is now *dead column* until a row-clear erases at least one of its cells.
  - Pre-fill cells exist; they are removed by row clears only.
  - Puzzle-style finite tray and target.
- Win condition: `boardMatchesTarget(board, target)`.
- Lose condition: tray non-empty AND `hasValidMoves(board, tray) === false`.
- New question per placement: "Does this placement steer toward completing a *row* (good — clear) or completing a *column* (bad — eight cells of dead weight that I now have to break later via a row clear that intersects them)?" Classic and Puzzle treat the two axes interchangeably; here they have opposite signs.
- Dual-purpose mechanic: **line completion (row vs column)**. The line-completion event is now sign-asymmetric: rows = sweep, columns = stuck-fill. Player wants completions on the row axis, fears completions on the column axis. Same primitive — `detectCompletedLines` — two senses gated by axis.
- Antagonist: dead columns the player accidentally completes; once stuck they restrict piece placements until eventually broken by a crossing row clear.
- Negative-space constraint: yes — empty cells must be preserved in at least one column per region the player intends to keep workable; killing the last empty cell of a column is a one-way trap.
- Order coupling: filling row R first then column C vs column C first then row R produces different histories of "stuck columns currently in play"; pieces that fit row-aligned vs column-aligned land in different orders.

## Candidate L1-5 — React: same-color adjacent cells are stable; different-color adjacencies erase both cells on placement

- Lens: 1 (Repurpose an existing primitive)
- Generated by: Claude Opus
- Rule changes:
  - On placement, after `placePiece` runs, scan each newly-placed cell's 4 orthogonal neighbors. If the neighbor is non-empty AND its color differs from the placed cell's color (and the neighbor is not part of this same placement), erase both cells (set to empty).
  - Resolve in a single deterministic pass (e.g. iterate placed cells in row-major order, mark cells to erase, then apply). No cascading reactions in the same turn (keeps work O(piece-cells × 4) ≈ O(20) per move).
  - Standard row/column clear runs *after* reactions on the resulting board.
  - Tray colors come from the existing 6-color palette; pieces keep their color across rotations.
  - Puzzle-style finite tray, target shows which cells must end filled (any color) and which must end empty.
- Win condition: `boardMatchesTarget(board, target)`.
- Lose condition: tray non-empty AND `hasValidMoves(board, tray) === false`.
- New question per placement: "Do I want adjacency reactions here? If yes, which neighbor cells am I willing to lose to remove the adjacent target/pre-fill cell? If no, which placement avoids any different-color contact?" Adjacency-color planning is wholly new — Classic/Puzzle ignore color entirely.
- Dual-purpose mechanic: **color**. Same primitive, two senses: same-color neighborhood = inert (cells stick) / different-color neighborhood = reactive (both cells erase). Player wants same-color contact for stable target construction *and* wants different-color contact for surgical deletion. The choice is the placement.
- Antagonist: pre-fill cells of a specific designated color. To remove them, the player must touch them with a piece of a *different* color; the cost is that the toucher cell also erases. So pre-fill removal is a 1-for-1 trade in cells.
- Negative-space constraint: yes — to preserve a target cell the player must surround it (or leave it unsurrounded) only by same-color or empty cells; a stray different-color neighbor wipes it out on the next adjacent placement.
- Order coupling: placing red-then-blue adjacently is symmetric to blue-then-red, but placing red-blue-red versus red-red-blue near the same site produces different chains of survival/erasure because the middle cell either survives or doesn't and changes what's adjacent next.

## Candidate L1-6 — Lifeline: tray does not auto-refill; line clears refill tray slots from a finite pre-shown queue, and only when slots are open

- Lens: 1 (Repurpose an existing primitive)
- Generated by: Claude Opus
- Rule changes:
  - Tray has 3 slots as today. Round-start populates slots 0..2 from a generator-built sequence of total length `N` (e.g. 7).
  - Auto-refill on tray-empty is *disabled*.
  - Line clear (row or column) consumes 1 piece from the refill **queue** and places it into the lowest-indexed empty tray slot. If no tray slot is empty when a clear fires, the queued piece is *lost* (popped from the queue and discarded).
  - Queue contents are visible (small status indicator above the tray showing the next 1–2 incoming pieces).
  - Pre-fill exists; target match win.
- Win condition: `boardMatchesTarget(board, target)`.
- Lose condition: tray empty AND queue empty AND target not met, OR no valid move on a non-empty tray.
- New question per placement: "Is this placement priming a clear? If so, will the tray have an empty slot at the moment the clear fires (so the queued refill lands), or will it waste the refill?" The player times clears against tray emptiness — neither dimension is a Classic concern.
- Dual-purpose mechanic: **line clear**. The clear is simultaneously a deletion event (erases cells, including pre-fill) and a tray-replenishment event. Player wants both — but the second only triggers when a slot is open, so clears must be timed against tray emptiness, and clears with a full tray *waste* the refill (a real loss).
- Antagonist: the queue draining to zero before the target is met. The queue is finite and visible; every wasted clear shortens the game. Pre-fill on the target is the other half of the antagonist (must be cleared, but every clear costs queue progression in addition to scoring it).
- Negative-space constraint: yes — leaving rows just barely incomplete, intentionally, until the tray drops to one piece, is the canonical move. Fully filling a row while three slots are full is a refill loss.
- Order coupling: place-clear-place-clear vs place-place-clear-clear refills different slots in different orders; same pieces, different surviving queues at end-game.

## Candidate L1-7 — Seed: every placement also writes one pre-fill cell at a deterministic anchor relative to the placed piece

- Lens: 1 (Repurpose an existing primitive)
- Generated by: Claude Opus
- Rule changes:
  - After `applyPlacementAndClear` finishes, compute the **seed cell**: the cell at `(row = origin.row + piece.height, col = origin.col + piece.width - 1)`, i.e. one cell directly below the piece's bottom-right corner; if out of bounds OR currently non-empty, the seed is suppressed.
  - The seed cell is filled with the pre-fill sentinel color. It does not retrigger clears (clears already ran).
  - Standard line clears apply on subsequent placements; seed cells clear like any pre-fill when their row/column is completed.
  - Puzzle-style finite tray and target.
- Win condition: `boardMatchesTarget(board, target)`.
- Lose condition: tray non-empty AND `hasValidMoves(board, tray) === false`.
- New question per placement: "Where will my placement's *seed* land — into a target cell I want filled (free progress), into a non-target cell I'll later have to clear (debt), into a non-empty cell (suppressed seed, free), or off the board (suppressed seed, free)?" Placement now has a side-effect cell whose location depends on the chosen origin and orientation.
- Dual-purpose mechanic: **placement event**. The placement primitive both writes the piece's footprint (own colored cells, possibly contributing to clears) and writes a 1-cell shadow of pre-fill at a piece-relative anchor. Player wants placements (board fill) and fears placements (each one breeds a pre-fill cell or wastes itself by suppression).
- Antagonist: the cumulative seed pre-fill the player created themselves. Plus an initial seed-pre-fill set from the generator. Player must defeat their own seed history.
- Negative-space constraint: yes — empty cells at piece-bottom-right anchors are now live "fuses"; placing a piece such that its anchor lands on an existing fill or the bottom edge is the only way to avoid debt.
- Order coupling: placing piece A at (1,1) then B at (3,3) seeds at (1+h_A, 1+w_A-1) and (3+h_B, 3+w_B-1); reversing the order may put A's anchor on top of B's footprint, suppressing A's seed. Same pieces, different seed totals.

## Candidate L1-8 — Resonance: two consecutive placements with matching cell-count erase both their footprints together (skipping the line-clear step that turn)

- Lens: 1 (Repurpose an existing primitive)
- Generated by: Claude Opus
- Rule changes:
  - State tracks `lastPlacedCellCount` (the piece-cells of the previous placement; null at round-start).
  - On placement, if `currentPiece.cells.length === lastPlacedCellCount`, mark a **resonance**: erase the cells of both the previous placement's footprint AND the current placement's footprint (set those cells to empty). Standard row/column clear detection is *skipped* this turn. Then set `lastPlacedCellCount = null`.
  - On non-resonance placements, run the standard clear pipeline as today, then set `lastPlacedCellCount = currentPiece.cells.length`.
  - Resonance erasure also removes pre-fill cells whose coordinates fall inside either footprint.
  - Puzzle-style finite tray and target.
- Win condition: `boardMatchesTarget(board, target)`.
- Lose condition: tray non-empty AND `hasValidMoves(board, tray) === false`.
- New question per placement: "Is the next piece I plan to place the same cell-count as the last one I placed? If yes, where do I aim *both* footprints so that the eventual resonance erases the pre-fill I want gone, without erasing the cells I need for the target?" The decision is now a *coupled pair*: previous and current placements jointly determine an erasure region.
- Dual-purpose mechanic: **piece cell-count** (which today only enters scoring). Same primitive, two senses: as a placement contribution it adds cells (good for filling target); as a *match-trigger* against the previous placement's cell-count it subtracts cells (good for clearing pre-fill, bad if it removes target cells). The player wants matching cell-counts at the moments where both footprints overlap pre-fill, and fears matching cell-counts elsewhere.
- Antagonist: pre-fill cells positioned such that no row/column completion can ever pass through them with the given tray (line-clear deletion is impossible) — they are *only* removable via resonance pairs.
- Negative-space constraint: yes — placed cells you intend to keep must not lie inside a future resonance footprint, so the previous placement's location effectively claims a "no-match-allowed" zone.
- Order coupling: the trigger condition itself is order-defined. Same multiset of pieces in order 3-3-4-4 produces two resonance pairs, in order 3-4-3-4 produces zero, and in order 4-3-4-3 produces zero. End-boards differ wildly.
