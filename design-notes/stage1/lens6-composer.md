# Lens 6 — Information variant (8 candidates)

Generated for Blockit design run. Lens: change what the player can see and when; unobserved state is part of the puzzle. Hidden information is deterministic and learnable (no pure-noise resolution).

---

## Candidate L6-1 — Veil-adjacent target mask (probe reveal)
- Lens: 6 (Information variant)
- Generated by: Composer-2-fast
- Rule changes: **vs Puzzle**: `puzzleTarget` is stored as usual, but a cell `SENTINEL_VEIL` is shown for any target cell not yet in `revealedTargetMask` (a bitmask over `BOARD_SIZE^2` in `GameState`). `revealedTargetMask` starts empty. `boardMatchesTarget` only evaluates against full `puzzleTarget` at win-check. Occupancy, non-target pre-fill, and "must be empty" for known non-target remain visible. Clear pipeline unchanged.
- Win condition: current board `boardMatchesTarget` with the stored full `puzzleTarget` (same as Puzzle: every target cell filled with correct color, every non-target cell empty).
- Lose condition: no legal placement exists (`!hasValidMoves`) and win predicate false; same as Puzzle.
- New question per placement: does this placement’s footprint include a cell 4-adjacent to a still-veiled target cell I care about, and does probing here waste shape needed later?
- Dual-purpose mechanic: every placement’s footprint is both a construction move and a **probe** that can expand `revealedTargetMask` for cells 4-adjacent to any placed cell (union update after place+clear).
- Antagonist: pre-fill and unknown target shape in veiled regions; veiled cells are the obscured part of the target pattern.
- Negative-space constraint: off-target empty remains mandatory where known; in veiled zones the player may mis-order commits without knowing the exact positive pattern.
- Order coupling: probe order changes which target cells are known when deciding later rotations and positions.
- Hidden info + reveal rule: which coordinates belong to the target (among cells still veiled) is **hidden** until a placed piece in that turn (after `applyPlacementAndClear`) has at least one cell 4-adjacent to that target coordinate, then that coordinate is marked revealed and its ring/duty shows true target membership.

## Candidate L6-2 — Sealed next-batch manifest (post-first-place unlock)
- Lens: 6 (Information variant)
- Generated by: Composer-2-fast
- Rule changes: **vs Puzzle** (tray is still 3 slots): the generator’s forward-sim precomputes `currentTray: PieceShape[3]` and `nextBatch: PieceShape[3]` (and optionally `nextNextBatch` if difficulty needs deeper lookahead) exactly as in `puzzleGenerator`’s “known future pieces” path, but the UI/overlay for `nextBatch` is **suppressed** until a flag `batchManifestUnlocked` flips. `SET_MODE` / `RESTART` reset `false`. After the first `PLACE_PIECE` in the instance that returns success, set `true` and render the multiset of `nextBatch` in the existing mode info chip (text or compact IDs). Undo may reset or keep unlock per design; default: unlock persists so O(board) per move.
- Win condition: same as Puzzle: `boardMatchesTarget` for the instance’s `puzzleTarget`.
- Lose condition: same as Puzzle: no legal moves while not won.
- New question per placement: with `nextBatch` now known, does the **order** in which I empty the current three slots to reach the next batch still serve line clears and the eventual target, or do I need to keep a “bridge” piece?
- Dual-purpose mechanic: the tray and refill pipeline both supply pieces to play **and** act as a clock: burning slots to empty advances toward the next known batch; batch timing is a resource with lookahead only after the first commit.
- Antagonist: pre-fill; the “future batch” is not negotiable, only *when* you face it.
- Negative-space constraint: where you park incomplete lines depends on you knowing upcoming shapes only after the first place (early game has less lookahead).
- Order coupling: different sequences of depletions change when `nextBatch` becomes active relative to the board.
- Hidden info + reveal rule: the exact multiset `nextBatch` (and optional deeper tail) is **hidden** in state until the first successful `PLACE_PECE` of the instance; from then on it is read-only public info in the chip. Generator remains forward-sim solvable; `nextBatch` is whatever the sim would draw next, fixed at construction.

## Candidate L6-3 — Active-slot peel: true shape on focus
- Lens: 6 (Information variant)
- Generated by: Composer-2-fast
- Rule changes: **vs Puzzle**: each tray index `i` has `peeled[i] : boolean` (start `false`). While `!peeled[i]`, the tray slot renders only a **count silhouette** (cell count) or a generic placeholder, not the true `PieceShape` geometry for legal placement/ghost, but `ROTATE_TRAY_PIECE` is still allowed and updates a hidden `piece[i]` in state. The first time in the round that `activeIndex` becomes `i` (focus/tap the slot to drag or keyboard cycle), set `peeled[i]=true` and the true shape + color are shown for that slot. `canPlacePiece` and ghost use true geometry only after peel for that slot. `PLACE_PIECE` is rejected if attempting to place from a slot that is not yet peeled. Generator unchanged; only presentation + gating. Reducer: O(1) extra per action.
- Win condition: Puzzle `boardMatchesTarget`.
- Lose condition: no legal **peel-aware** move (if `hasValidMoves` is computed using true shapes only for peeled slots, this matches “must inspect before place”); standard Puzzle no-move loss when the position is lost.
- New question per placement: which slot do I **activate first** to reveal geometry before I commit a global placement order across the batch?
- Dual-purpose mechanic: the tray “active slot” both selects which piece to place and is the **inspection** action that pays an attention cost in move order.
- Antagonist: pre-fill; the “unknown geometry while unpeeled” is information pressure, not a new piece shape.
- Negative-space constraint: you may delay peeling a large piece to keep search smaller, at the risk of discovering a bad fit too late.
- Order coupling: peel order is independent of play order; both decide what information is available for later commits.
- Hidden info + reveal rule: the true `PieceShape` and chroma for each slot is **hidden** until that slot is first `activeIndex` in the instance; then `peeled[i]` locks visible truth for the rest of the round (until refill clears peel flags if you clear tray in same run).

## Candidate L6-4 — Column-curtain pre-fill (target always lit)
- Lens: 6 (Information variant)
- Generated by: Composer-2-fast
- Rule changes: **vs Puzzle**: maintain `curtain: boolean[8]`, all `true` at `RESTART`. For column `c` with `curtain[c]`, any cell `(r,c)` that is **not** in `puzzleTarget.keys` (non-target) renders as `SENTINEL_CURTAIN` instead of the actual occupancy; target-ring cells in that column still show the target ring and are not curtained. When `curtain[c]` is `true`, `canPlacePiece` for placements that would read occupied/empty under curtain uses **true** board state in reducer (curtain is visual/information, not physical); alternatively hard rule: `canPlacePiece` consults true board, consistent with O(board). After each successful `PLACE_PIECE`, for every column that received ≥1 new occupied cell in the placed footprint (after place, before or after clear per spec—pick **after** `applyPlacementAndClear`), set `curtain[c]=false` for that column. Once false, that column’s non-target cells show true pre-fill/empty. Win-check uses true board, not display.
- Win condition: `boardMatchesTarget` on true occupancy.
- Lose condition: no legal move on true board, not won.
- New question per placement: in which **column** should I first spend shape to uncurtain hidden junk, given only target is visible everywhere?
- Dual-purpose mechanic: every placement in column c both changes occupancy **and** can lift the curtain for that column (information gain).
- Antagonist: pre-fill in curtained non-target space; the curtain is the obscured part of the initial board.
- Negative-space constraint: you must not fill cells that are actually empty non-target, but you cannot see that under curtain without probing that column.
- Order coupling: column probe order changes when you can see clumps; same moves in different column order yields different *visible* information mid-game, not different physics.
- Hidden info + reveal rule: non-target occupancy in column `c` is **hidden** (curtained) while `curtain[c]`; **reveals** the moment `curtain[c]` flips to `false` after a placement with footprint intersecting that column (post-clear state).

## Candidate L6-5 — Quadrant-sealed target submask
- Lens: 6 (Information variant)
- Generated by: Composer-2-fast
- Rule changes: **vs Puzzle**: index quadrants as `Q ∈ {0,1,2,3}` for top-left, top-right, bottom-left, bottom-right 4×4 blocks. Maintain `quadrantRevealed: boolean[4]`, all `false` at start. While `!quadrantRevealed[Q]`, target cells whose coordinates lie in Q show as SENTINEL_VEIL (or generic “unknown target” ring) with no per-cell on/off. `boardMatchesTarget` requires full `puzzleTarget` at end. `quadrantRevealed[Q]=true` when a placed piece (after `applyPlacementAndClear`) has ≥1 cell with coordinates in Q. Revealed Q shows exact per-cell target requirements for that quadrant.
- Win condition: `boardMatchesTarget` with full `puzzleTarget` at end.
- Lose condition: no legal move, not won.
- New question per placement: which **quadrant** do I enter first to unseal, given cross-quadrant pieces and the need to clear pre-fill in sealed zones?
- Dual-purpose mechanic: every placement is both build toward a quadrant’s eventual pattern **and** a key that unlocks that quadrant’s exact target submask.
- Antagonist: pre-fill; sealed quadrants hide which target positives exist in that block until entry.
- Negative-space constraint: you must still respect global empty, but the sealed quadrant’s *positive* pattern is unknown until a footprint touches it.
- Order coupling: entering quadrants in different orders changes what you know for mid-game line clears in other quadrants.
- Hidden info + reveal rule: per-quadrand boolean **membership in `puzzleTarget` for cells in that 4×4** is **hidden** until a placement’s footprint (post-clear) intersects that quadrant; then the entire quadrant’s target submask is revealed at once.

## Candidate L6-6 — First-clear unmasks forbidden template
- Lens: 6 (Information variant)
- Generated by: Composer-2-fast
- Rule changes: **vs Puzzle**: define `K` fixed templates `F_0,…,F_{K-1}` (sets of 2–4 coordinates, precomputed), disjoint from the target’s positive support in that instance. At puzzle build, the generator draws `forbiddenIndex ∈ {0,…,K-1}` deterministically from the instance seed, proves via forward-sim that some solution path exists that ends with `boardMatchesTarget` and zero occupancy on `F[forbiddenIndex]`, and stores `forbiddenIndex` in `GameState` (not shown in UI). The player’s state has `forbidden: Coord[] | null = null` until a line clear (row or column) first occurs in the **player** session; on that event, set `forbidden = F[forbiddenIndex]` and render those coordinates as sentinels as “taboo to occupy at end.” If `forbidden` is set and the final board occupies any cell in `forbidden`, lose. `boardMatchesTarget` is unchanged. Generator should reject any candidate with no first-clear in the sim if that edge case is awkward; otherwise a fallback is to reveal taboo on first successful place (design choice).
- Win condition: `boardMatchesTarget` **and** no cell in `forbidden` is occupied on the final board (when `forbidden` is non-null before final place; or vacuously if never revealed—generator should always force reveal before win in practice).
- Lose condition: no legal move without win, **or** final board violates taboo-occupation when `forbidden` is set.
- New question per placement: before the first clear, which placements preserve **some** first-clear line that leaves a still-feasible path to target without ever needing `F[forbiddenIndex]` occupied (you infer threat only after reveal; after reveal, the question is route under known taboo)? After reveal: each placement is constrained by a **second** set beyond target—same as a learnable “extra negative” once lit.
- Dual-purpose mechanic: the first line clear is both standard Puzzle’s deletion tool **and** the trigger that discloses the fixed taboo template for this instance.
- Antagonist: pre-fill; after reveal, `F` is an on-board constraint set the player must route around; before reveal, uncertainty is which `F` it will be (finite menu from legend).
- Negative-space constraint: you must end empty on `F`; planning must hedge possible `F` pre-reveal, then use exact `F` post-reveal.
- Order coupling: moves before first clear are taken under a **small** ensemble of possible `F`; after first clear, order is the usual target + taboo joint feasibility.
- Hidden info + reveal rule: the index `forbiddenIndex` (hence which `F_k`) is **hidden** at round start; **reveal** on the first line-clear in the play session, then show `F[forbiddenIndex]`. The index is **not** re-rolled on clear—it is **fixed** at build (deterministic, learnable: over many games, you learn the template shapes and the winnowing rule the generator used to make each index solvable, not a noise draw at reveal).

## Candidate L6-7 — Cleared-line target illumination (per-axis visibility)
- Lens: 6 (Information variant)
- Generated by: Composer-2-fast
- Rule changes: **vs Puzzle**: maintain `rowSeen: boolean[8]`, `colSeen: boolean[8]`, all `false`. A target cell at `(r,c)` is only drawn in the target overlay in full (required color) if `rowSeen[r] || colSeen[c]`. If neither, show `SENTINEL_VEIL` for that target cell (no color commitment visible). `boardMatchesTarget` on reducer still uses full `puzzleTarget` bitset. `rowSeen`/`colSeen` set `true` when a line clear of that **row** or **column** occurs on that index (row clear sets `rowSeen[r]`, col clear sets `colSeen[c]`; a double clear sets both for affected lines). No change to `applyPlacementAndClear` beyond flags after clear.
- Win condition: `boardMatchesTarget` (full).
- Lose condition: no legal move, not won.
- New question per placement: which line should I try to clear **now** to illuminate target on that axis, versus placing to set up, given partial visibility of which colors where?
- Dual-purpose mechanic: line clears are both the Puzzle delete for pre-fill **and** the only way to light target requirements along that row/column in the UI.
- Antagonist: pre-fill; dark target (unknown which cells/colors) until the corresponding axis has cleared at least once.
- Negative-space constraint: you may place “blind” on dark axes, risking a wrong final color in a cell you could not read yet—generator must only emit fully solvable instances under the visibility rule, or the player must infer from cross-illumination.
- Order coupling: clear order changes which target cells are visible for subsequent commits.
- Hidden info + reveal rule: for each target cell `(r,c)`, the fact that it is a target and its required color is **hidden in UI** until `rowSeen[r] || colSeen[c]`; **reveals** on first clear of row `r` or column `c` (after clear animation tick).

## Candidate L6-8 — Kicker table disclosure on slot-rotate (deterministic +1)
- Lens: 6 (Information variant)
- Generated by: Composer-2-fast
- Rule changes: **vs Puzzle**: for each `piece.id` in `PIECE_CATALOG` and each `rotation R ∈ {0,1,2,3}` in the piece’s local space, a fixed public table `KICK[dedupeKey(id)][R] → (dr,dc)` in piece-local “footprint up” space gives one extra monomino cell in board space after the piece is rotated. While placing, the legal footprint is the union of the canonical placed cells and the anchor+rotated kicker. **Hidden in tray**: the kicker is not included in the tray ghost until the player has called `ROTATE_TRAY_PIECE` on that active slot at least **once in that slot’s current “cycle”** (new tray batch resets a `rotateCount[i]` to 0; first rotate sets `kickerDisclosed[i]=true`). For generator forward-sim, the sim’s placements include kicker. If kicker would overlap filled or violate bounds, the placement is illegal (rejected in `canPlacePiece`). `canPlacePiece` and ghost use the union only after kicker is disclosed, OR always use union internally but the **player** only *sees* the kicker after first rotate (information-only gating) — the former avoids blind illegal tries; the latter is fairer: **reveal** after one rotate, **validation** always uses kicker. Preferred: `canPlacePiece` always includes kicker; only **preview** lags. Then no blind failure; still information variant: you must rotate once to *see* the kicker. Actually user asked hidden info: “piece’s true shape is revealed only after you commit a where + rotation” is one sketch — this is kicker, not new shape, from same catalog.
- Win condition: `boardMatchesTarget` on final board.
- Lose condition: no legal placement.
- New question per placement: after the first rotate on this slot, which `(origin, final rotation)` makes the kicker sit on a cell that will help/hurt clears and target, given the table is learnable?
- Dual-purpose mechanic: `ROTATE_TRAY_PIECE` is both orientation choice **and** the disclosure action for the kicker cell for the active slot in this batch.
- Antagonist: pre-fill; kicker can accidentally complete or block lines.
- Negative-space constraint: the kicker may deny placements that look legal from base silhouette alone; empty cells matter around the +1.
- Order coupling: which slot is rotated (for disclosure) before drag affects planning across the three pieces.
- Hidden info + reveal rule: the kicker offset for `(active piece, current rotation)` is **hidden from the board ghost and tray art** until `rotateCount[activeIndex] ≥ 1` since the current piece entered the slot; **reveals** on first rotate. Table `KICK` is fixed and can be in the mode legend; not random.

## Summary

Eight information-variant mode sketches are listed (L6-1…L6-8). They vary **what** is hidden (target topology, next batch, true tray geometry, pre-fill in columns, quadrant masks, a taboo template index fixed at build, per-axis target color visibility, a deterministic kicker cell) and **when** it reveals (4-adjacency, first place, first focus, first footprint in a column, first footprint in a quadrant, first line clear, first line clear of a given row/column, first rotation). All avoid pure post-hoc random resolution (Scar disease); L6-6’s taboo is fixed at instance creation and only **disclosed** on first clear (not re-rolled) so the player can learn the finite template set; none remove tray choice (Pipeline) or add a win sub-condition that is implied by the others alone; each adds a **post-reveal** or **legibility** constraint the player can still violate. Implementation paths assume a `mode === 'l6-*'` (or a single `informationVariant: enum`) branch in `gameReducer.ts`, O(board) or O(1) extra flags per move, forward-sim generation with visibility/taboo invariants, and existing sentinel-styled cells.
