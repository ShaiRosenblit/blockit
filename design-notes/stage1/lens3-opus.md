# Lens 3 — Couple two currently-independent decisions (Claude Opus)

Lens recap: in Classic the per-move decisions are (slot, rotation, origin). They are independent within a move and across moves, except for "does it fit?". Each candidate below introduces a coupling between two currently-independent decisions such that the player must reason about them **jointly**, while preserving (or adding) total agency.

Anti-pattern screen applied to all 8:
- **Mirror**: every coupling leaves both legs free over a multi-element legal set; neither leg is a deterministic function of the other.
- **Breathe**: no candidate adds a win sub-condition that's implied by the others.
- **Pipeline**: every candidate adds at least one new joint decision dimension; restrictions on existing decisions are compensated by routing/ordering decisions.
- **Scar**: every coupling is deterministic and player-controllable.

## Candidate L3-1 — Slot-axis clear contract
- Lens: 3 (Couple two currently-independent decisions)
- Generated by: Claude Opus
- Rule changes: Tray slot index 0 is permanently labeled `ROW`, slot 1 is `COL`, slot 2 is `FREE` (labels are properties of the slot UI position, not of pieces; refills inherit slot labels). A placement using slot 0 is rejected unless `applyPlacementAndClear` would clear ≥1 row this turn; slot 1 is rejected unless it clears ≥1 column; slot 2 is unconstrained. The placement preview turns red and the move is illegal if the contract isn't met. Pre-fill seeded as in Puzzle. Tray refills as a 3-batch when all slots are empty (Classic refill, NOT Pipeline's round-robin).
- Win condition: board occupancy exactly matches `puzzleTarget` (puzzle-style).
- Lose condition: tray non-empty AND no `(slot, rotation, origin)` triple satisfies its slot's contract while passing `canPlacePiece`.
- New question per placement: "Which slot's contract can I satisfy using the current near-complete lines, and in which order do I spend my three slots so each one's clear-axis contract becomes satisfiable when its turn comes?" The slot decision (was independent of clear-axis) is now coupled with which axis the player intends to clear.
- Dual-purpose mechanic: line clears. Want — clears delete pre-fill (Puzzle's repurposing) AND legalize slot 0/1. Fear — completing a row with the slot 1 piece "wastes" the row toward slot 1's contract; completing a column with the slot 0 piece is illegal.
- Antagonist: slot-locked board states — pre-fill arrangements with no near-complete row force the player to use slot 2 or slot 1 first to manufacture row-completability for slot 0.
- Negative-space constraint: non-target cells must end empty (puzzle constraint).
- Order coupling: very strong. The same three pieces played `[FREE, ROW, COL]` vs `[ROW, FREE, COL]` give different clears (the FREE play sets up which row/col each contract slot can later complete), so the per-batch slot-ordering decision is now load-bearing.

## Candidate L3-2 — Origin-row parity must match rotation parity
- Lens: 3 (Couple two currently-independent decisions)
- Generated by: Claude Opus
- Rule changes: Each tray piece carries a rotation index `k ∈ {0,1,2,3}` already implicit in the rotate-tray pipeline. Each board row 0..7 is labeled by row-parity (even rows → "even-rot"; odd rows → "odd-rot"). A placement is legal only if `(piece.rotationIndex mod 2) == (origin.row mod 2)`. Tap-to-rotate still cycles `k` through all 4 indices, so every piece can reach both parities; rotation-equivalent shapes (square 2×2, plus-pentomino) still cycle `k` in state even if their cells don't change, and the parity rule still applies to that index. Pre-fill seeded as in Puzzle.
- Win condition: board occupancy exactly matches `puzzleTarget`.
- Lose condition: tray non-empty AND no `(rotationIndex, origin)` exists for any tray piece such that parity matches AND `canPlacePiece` succeeds.
- New question per placement: "For this piece, which rotation-index × origin-row pairing satisfies parity AND fits AND lands its cells inside the target?" Two of the four rotation indices are gated out by the chosen row, and four of the eight rows are gated out by the chosen rotation index — but neither leg fully determines the other (each parity bucket has 2 rotations and 4 rows), so the player must coordinate them.
- Dual-purpose mechanic: rotation. Want — rotate to access the row you need (rotation now selects half the board's rows). Fear — rotating to fix a near-target column may push you out of the row whose pre-fill you needed to clear.
- Antagonist: parity-banded pre-fill — pre-fill clumps along same-parity rows force the player to commit to one parity for several moves, then cleared rows shift planning.
- Negative-space constraint: non-target cells must end empty.
- Order coupling: clearing a row removes pre-fill from a specific row (still indexed by its original parity), so a clear can free up an "odd-rot" row for a later odd-parity rotation; the order in which you commit to even vs odd parity moves changes which target cells you can still reach.

## Candidate L3-3 — Bounding-box must share a row or column projection with previous placement
- Lens: 3 (Couple two currently-independent decisions)
- Generated by: Claude Opus
- Rule changes: After every placement, the placement's bounding box `{r0..r1, c0..c1}` is stored as `lastBBox`. The next placement is legal only if its bounding box's row-projection `[r0..r1]` overlaps `lastBBox` row-projection by ≥1 row OR its column-projection overlaps by ≥1 column. The first placement of a puzzle is unconstrained. `lastBBox` resets when the tray refills mid-batch? No — it persists across refills. Pre-fill seeded as in Puzzle. The previous bounding box is rendered as a faint outline on the board so the constraint is visible.
- Win condition: board occupancy exactly matches `puzzleTarget`.
- Lose condition: tray non-empty AND no `(slot, rotation, origin)` exists that both passes `canPlacePiece` and satisfies the bbox-projection-overlap rule.
- New question per placement: "Given exactly where I just placed (bbox), which of my three pieces can I place where the new bbox shares a row or a column with the old bbox AND advances the target — and which side (row-shared vs column-shared) should I extend to keep next turn's options open?" The placement decision and the prior placement decision are now joined.
- Dual-purpose mechanic: bounding box. Want — extend the bbox over the target so subsequent moves are constrained to useful regions. Fear — a row-only chain pulls all future placements into the same horizontal band, narrowing options.
- Antagonist: corner-trapping pre-fill — pre-fill that forces the previous placement against an edge, restricting the projection-overlap region for the next move.
- Negative-space constraint: non-target cells must end empty.
- Order coupling: maximal — the legal placement set at turn T is a strict function of where exactly turn T-1's piece landed. Re-ordering the same three pieces yields entirely different reachable sets.

## Candidate L3-4 — Color-typed targets, mismatches must be cleared
- Lens: 3 (Couple two currently-independent decisions)
- Generated by: Claude Opus
- Rule changes: Three-color palette `{A, B, C}`. Tray refills always deliver one piece of each color (color is a slot-position property of the refill batch, like Puzzle's pre-shuffled tray). `puzzleTarget` becomes `TargetCell[][]` where each target cell carries a required color in `{A,B,C}`. `canPlacePiece` is unchanged. After placement, each placed cell that lands on a target cell with **matching** color is marked correctly-filled; on a **mismatched-color** target it occupies the cell normally but is flagged `wrong`. Row/column clears erase whatever's in the line (correct, wrong, or pre-fill alike) — clearing is the only way to remove a `wrong` cell from a target position.
- Win condition: every target cell ends correctly-filled (i.e., currently occupied by a block whose color matches the cell's required color), AND every non-target cell is empty.
- Lose condition: tray non-empty AND no `(slot, rotation, origin)` exists that passes `canPlacePiece` (placement-feasibility rule unchanged); also lose if tray empty and target has a `wrong`-stamped or empty target cell with no remaining clears reachable.
- New question per placement: "Which slot (= which color) AND which origin together stamp the most matching-color cells onto the target — and is the wrong-color collateral I'm about to stamp clearable later via a row/column?" Slot and origin were previously joined only by 'does it fit'; now they are joined by 'does the color overlay align'.
- Dual-purpose mechanic: line clears. Want — clear a row to erase a wrong-color stamp blocking a target cell. Fear — that same clear erases correct-color cells in the row, forcing re-fill with the right slot's color.
- Antagonist: pre-stamped wrong-color cells in target positions (generator can seed them on the start board), which the player must overcome via clears.
- Negative-space constraint: non-target cells must end empty (Puzzle constraint, inherited).
- Order coupling: clears change which colored cells survive in the target; deliberately wrong-stamping early to fix later via clear is order-sensitive, and the order in which you spend your color-typed slots within a batch determines which target color goes where first.

## Candidate L3-5 — Beacon-adjacency chain, beacon moves to placement centroid
- Lens: 3 (Couple two currently-independent decisions)
- Generated by: Claude Opus
- Rule changes: A single `beacon: Coord` is stored in state and rendered on the board as a marker dot (no occupancy effect — pieces may be placed on the beacon's cell). Generator picks a starting beacon position. A placement is legal only if at least one of its placed cells is at Manhattan distance 1 from the beacon's current cell (i.e., adjacent — the cells themselves can include the beacon cell, but at least one cell of the piece must be a 4-neighbor of the beacon cell). After a successful placement, the beacon moves to `round(centroid(placedCells))`, clamped to the nearest in-bounds cell. `canPlacePiece` is unchanged.
- Win condition: board occupancy exactly matches `puzzleTarget`.
- Lose condition: tray non-empty AND no `(slot, rotation, origin)` exists that satisfies the beacon-adjacency rule AND `canPlacePiece`.
- New question per placement: "Where can my piece both touch the beacon's neighborhood AND advance the target AND end with its centroid in a position from which my next piece will still have a legal beacon-adjacent placement?" The current placement decision is coupled with the beacon's history (a function of all previous placements), and the player is also choosing where the beacon will be next turn (centroid steering).
- Dual-purpose mechanic: the beacon. Want — keep the beacon near unfilled target cells so future placements have legal adjacencies into the target. Fear — large-bounding-box pieces drag the beacon centroid into corners or onto already-filled regions, stranding the next placement.
- Antagonist: pre-fill ringing the beacon's start position, forcing the player's first placements into specific shapes that route the beacon away from the target.
- Negative-space constraint: non-target cells must end empty.
- Order coupling: the legal placement set at turn T equals "cells adjacent to centroid(placement T-1)"; reordering the tray rewrites the entire path of beacon positions through the puzzle.

## Candidate L3-6 — Bounding-box dimensions transposed from previous
- Lens: 3 (Couple two currently-independent decisions)
- Generated by: Claude Opus
- Rule changes: Track `lastBBoxDims: {w: number; h: number} | null`. A placement is legal only if `(currentBBox.w == lastBBoxDims.h) AND (currentBBox.h == lastBBoxDims.w)` — i.e., the new bounding box is the transpose of the previous one (a 3×2 must be followed by a 2×3; a square must be followed by a square of any size; a 5×1 must be followed by a 1×5). First placement is unconstrained. Tap-to-rotate cycles the piece's stored rotation index through all 4 indices regardless of geometric symmetry; only post-rotation `(width, height)` matters for the rule. `canPlacePiece` is unchanged.
- Win condition: board occupancy exactly matches `puzzleTarget`.
- Lose condition: tray non-empty AND no `(slot, rotation, origin)` exists with transposed dims AND `canPlacePiece`. Tray with three squares-in-a-row remains feasible because squares chain to squares; tray that goes `square → 5×1` is dead — the generator must avoid such sequences during forward-sim.
- New question per placement: "Of my three tray pieces, which can I rotate to match the required `H×W`, AND where does that rotation fit, AND will my chosen rotation's NEW `W×H` leave a transposed shape I can serve next turn from the remaining tray?" Rotation choice and previous-bbox state are now coupled; piece selection is also coupled (some pieces simply can't satisfy certain dim contracts).
- Dual-purpose mechanic: rotation. Want — rotate to match the dimension contract. Fear — the same rotation that satisfies this turn's contract is the wrong shape for the target cells you need to fill.
- Antagonist: dimension dead-ends — sequences of bbox shapes that force the player into a `W×H` with no remaining tray piece able to provide `H×W`. Generator forward-sims until a full chain is feasible.
- Negative-space constraint: non-target cells must end empty.
- Order coupling: maximal across the tray — playing the 4×2 piece after a 2×4 vs after a 4×2 produces entirely different chains. The same three pieces in a different order can be unsolvable.

## Candidate L3-7 — Twin patterns, every placement must touch both
- Lens: 3 (Couple two currently-independent decisions)
- Generated by: Claude Opus
- Rule changes: There are two target patterns, `targetA: boolean[][]` and `targetB: boolean[][]`, drawn in distinct colors (A in cyan ring, B in magenta ring). The patterns may overlap in space. A placement is legal only if at least one of its placed cells lands on a `targetA == true` cell AND at least one lands on a `targetB == true` cell. (A single placed cell that lies on a both-A-and-B target cell counts once for each.) `canPlacePiece` unchanged. Pre-fill optional but small.
- Win condition: every `targetA` cell occupied AND every `targetB` cell occupied AND every cell outside `targetA ∪ targetB` empty.
- Lose condition: tray non-empty AND no `(slot, rotation, origin)` exists that passes `canPlacePiece` AND covers ≥1 cell of each pattern.
- New question per placement: "For each of my three pieces, which `(rotation, origin)` jointly stretches the piece across both A and B AND advances at least one of them toward completion?" Origin and rotation are now joined by a dual-coverage requirement instead of just 'fits'.
- Dual-purpose mechanic: pattern overlap. Want — pieces whose shape spans the gap between A and B regions cover lots of ground. Fear — a piece that can only span A-and-B at one specific origin is forced into that origin even if it stamps wrong cells in non-target space (which then must be cleared).
- Antagonist: spread patterns — generator places A and B regions far enough apart that only longer pieces or specific rotations can span both.
- Negative-space constraint: non-`(A ∪ B)` cells must end empty (the negative space is the union's complement, not just one pattern's complement, so the player must clear away spillover from spanning placements).
- Order coupling: clears erase from both A-cells and B-cells in the line, so a row-clear that resets one pattern also resets the other; deliberately overshooting into A on early moves to clear later (resetting both) is sometimes necessary, and the order matters because each placement must touch both even when one is already saturated.

## Candidate L3-8 — Per-slot contract carousel rotates each turn
- Lens: 3 (Couple two currently-independent decisions)
- Generated by: Claude Opus
- Rule changes: Score-attack mode (no target). Three contracts are defined: `α` = "placement causes ≥1 line clear this turn"; `β` = "placement's bounding box fits inside columns 0..3"; `γ` = "placement's bounding box fits inside columns 4..7". State stores `slotContracts: [α|β|γ, α|β|γ, α|β|γ]` initialized as `[α, β, γ]`. A placement using slot `S` is legal only if `slotContracts[S]` is satisfied (canPlacePiece must also pass). After every placement, contracts cycle one position right: `slotContracts = [slotContracts[2], slotContracts[0], slotContracts[1]]`. Tray refills by Classic 3-batch rule (refills don't reset the carousel).
- Win condition: score reaches the difficulty's target threshold (Classic-style score-attack).
- Lose condition: tray non-empty AND no `(slot, rotation, origin)` exists that passes `canPlacePiece` AND its slot's currently-assigned contract.
- New question per placement: "Of the three pieces I have right now, which order maps each piece to a contract its shape can satisfy — and is it worth playing a 'free' piece into a hard contract now to leave the easier contract aligned with my best piece next turn?" The slot decision (was independent of turn-index) is now coupled with `turnIndex mod 3`, because contract-on-slot-S depends on how many placements have occurred.
- Dual-purpose mechanic: slot index. Want — pick the slot whose current contract matches your best piece's natural placement. Fear — the contract assigned to that slot next turn (after the cycle) must still be satisfiable by whichever piece you didn't play; mis-timing strands a piece against a contract its shape can never meet.
- Antagonist: contract-shape mismatches in the tray — e.g., a 5×1 horizontal I-piece can never satisfy `β` (cols 0..3 only) at any origin because its width-5 bbox doesn't fit; the player must either rotate it before playing under `β`, or sequence the carousel so the I-piece lands on `α` or `γ`.
- Negative-space constraint: none structural; standard Classic accumulation pressure.
- Order coupling: the same 3-piece batch played in order `[0,1,2]` vs `[2,0,1]` exposes each piece to a completely different contract sequence, so per-batch ordering — which had only weak clear-coupling in Classic — becomes the central decision.
